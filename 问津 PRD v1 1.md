## 1. 项目概述

### 1.1 一句话定位

一个 AI 驱动的、任何课程老师都能一键搭建的智能学习导航平台：老师导入课程即可生成知识图谱，学生借助它完成学情诊断、获取个性化学习路径与 AI 学习建议。

### 1.2 解决的核心问题

传统课程学习中存在三个长期痛点：

1. **学生不知道自己”卡在哪、为什么卡”。** 一道题做错，学生往往只知道”这题不会”，却不知道真正的薄弱根因在更底层的前置知识点上。
1. **学习缺乏个性化路径。** 所有学生面对同一套教材按同一顺序学习，无法根据个人掌握情况动态调整”接下来该学什么”。
1. **知识图谱构建门槛高。** 个性化学习的前提是知识图谱，而梳理知识点之间的依赖关系是高度耗费人力的人工工作，导致大多数同类系统只能停留在单门写死的课程上，无法推广。

本系统针对性地解决以上三点，其中第 3 点（知识图谱构建难）正是同类项目公认的最难环节，本项目通过”AI 辅助建图谱 + 标准化导入”将这一难点转化为产品亮点。

### 1.3 产品类比与命名

可类比”学习版的导航软件”：导航软件需要知道”你在哪（定位）、要去哪（目标）、怎么走（路径）”，并在情况变化时”实时重新规划”。本系统对学习过程做同样的事——定位学情、诊断根因、规划路径、动态调整。

「问津」之名即由此而来：学生面对一门课”不知卡在哪、不知接下来学什么”，正是身处迷津；系统所做的一切——诊断、回溯、导航——本质上是一件事：**指点迷津**。

### 1.4 目标用户

|用户角色    |核心诉求                  |对应功能入口           |
|--------|----------------------|-----------------|
|学生      |知道自己学得怎么样、接下来该学什么、卡住能问|学情诊断、学习路径、AI 学习伴侣|
|教师      |低成本搭建课程知识图谱、了解学生整体学情  |图谱构建、学情概览        |
|管理人员（远期）|跨课程的教学数据洞察            |后台统计（远期规划）       |

### 1.5 本版范围声明

- **做深**：软件工程一门课的学生端完整体验（诊断、路径、自适应、AI 伴侣）全部跑通。
- **架构预留**：数据库与系统架构从设计之初即支持多课程（多租户、`course_id` 隔离），不为后续扩展返工。
- **平台能力（B 档）**：教师端实现”结构化文件导入 + 可视化微调”；“AI 自动建图谱”做到可演示验证的程度，作为核心技术亮点展示。
- **远期规划**：多课程规模化运营、管理端跨课程统计、移动端深度适配等，写入演进规划，不在本版交付。

## 2. 系统总体架构

系统自上而下分为四层，数据在层间单向流动并形成闭环。

### 2.1 四层结构

```
┌─────────────────────────────────────────────┐
│  ① 图谱构建层（教师端）                          │
│     AI 自动建图谱 / 结构化导入 / 可视化微调        │
└───────────────────┬─────────────────────────┘
                    │ 统一数据契约（点+边 JSON）
┌───────────────────▼─────────────────────────┐
│  ② 数据与存储层                                 │
│     知识图谱（节点表+边表）/ 多课程隔离 / 学情数据   │
└───────────────────┬─────────────────────────┘
                    │
┌───────────────────▼─────────────────────────┐
│  ③ 智能核心层                                   │
│     学生画像 / 认知诊断 / 动态路径 / AI 学习伴侣    │
└───────────────────┬─────────────────────────┘
                    │
┌───────────────────▼─────────────────────────┐
│  ④ 体验呈现层（学生端）                          │
│     知识地图 / 学习路径 / 成长档案 / 资源匹配       │
└─────────────────────────────────────────────┘
```

### 2.2 统一数据契约（关键设计）

无论知识图谱来自 AI 自动生成、结构化文件导入，还是手动编辑，最终都收敛为同一套内部数据结构（节点 + 边）。这一设计保证了三种构建方式殊途同归，架构干净，也是技术方案的核心亮点之一。

内部统一格式示意：

```json
{
  "course": "软件工程",
  "nodes": [
    {"id": "K01", "name": "软件工程概述", "chapter": "第1章", "difficulty": 1, "description": "学科基本概念"},
    {"id": "K10", "name": "软件过程模型", "chapter": "第2章", "difficulty": 2, "description": "过程模型总览"},
    {"id": "K12", "name": "瀑布模型", "chapter": "第2章", "difficulty": 2, "description": "经典线性过程模型"}
  ],
  "edges": [
    {"from": "K01", "to": "K10", "type": "前置"},
    {"from": "K10", "to": "K12", "type": "包含"}
  ]
}
```

## 3. 图谱构建层（教师端）

### 3.1 功能目标

让教师以尽可能低的人力成本，为任意课程构建出一张准确的知识图谱。这是整个系统的地基，也是把项目从”单课程作业”升级为”可复制平台”的关键。

### 3.2 三种构建方式

**（1）AI 自动建图谱（核心亮点，B 档：做到可演示验证）**

- 教师上传课程教学大纲 / 教材目录 / 课程 PPT。
- 系统调用大模型，在教师提供的材料范围内抽取知识点、推断知识点之间的依赖关系。
- AI 输出直接对齐统一数据契约（点 + 边 JSON），生成知识图谱初稿。
- 教师进入可视化界面对初稿进行人工审核与微调。
- **质量验证指标（v1.1 新增）**：以教师人工修正后的图谱为基准（Ground Truth），统计 AI 初稿的两项指标——**节点召回率**（AI 找全了多少知识点）与**边准确率**（AI 推断的依赖关系有多少被保留）。答辩时以量化数据呈现（如”节点召回率 90%+，教师仅微调约 10% 的边”），将”AI 辅助建图谱”从口号变为有数据支撑的结论。该数据在构建软件工程演示图谱的过程中即可顺手采集，无额外成本。

> 设计要点：AI 的价值不在于”建图谱”这个技术动作（存两张表即可），而在于解决”知识点之间到底什么关系”这一最难的人工环节。AI 生成初稿，人工修正，是效率与质量的最佳平衡。

**（2）结构化文件导入（务实兜底，必做）**

- 提供标准化 Excel / JSON 模板，教师按格式填写后一键导入。
- Excel 模板含两张表：知识点表（节点）与依赖关系表（边）。详见 3.3。

**（3）可视化微调与手动编辑（必做）**

- 在 ECharts 图上以拖拽方式增删知识点、增删/修改关系。
- 用于修正 AI 生成结果，或零散手动构建。

### 3.3 结构化导入模板设计

**知识点表（节点）字段：**

|字段   |说明                          |示例      |
|-----|----------------------------|--------|
|知识点ID|唯一标识，关系表通过 ID 关联（不用名称，避免脏数据）|K12     |
|知识点名称|显示用                         |瀑布模型    |
|所属章节 |章节归属                        |第2章     |
|难度   |数字分级 1–5（便于参与计算，不用文字）       |2       |
|描述   |简要说明                        |经典线性过程模型|

**依赖关系表（边）字段：**

|字段  |说明               |示例 |
|----|-----------------|---|
|起点ID|关系起点             |K10|
|终点ID|关系终点             |K12|
|关系类型|枚举值（下拉选择，不允许自由填写）|包含 |

**关系类型枚举（统一口径，禁止自由发挥）：**

|类型|含义        |用途       |
|--|----------|---------|
|前置|学完 A 才能学 B|路径推荐的核心依据|
|包含|章节包含知识点   |结构组织     |
|相关|易混淆 / 相关联 |辅助提示     |
|应用|知识点对应典型题型 |题目关联     |

### 3.4 导入校验规则（体现工程严谨性）

导入时系统必须执行校验并给出可定位的友好报错，而非直接失败：

- **孤立 ID**：边表引用了节点表中不存在的 ID（如”第 5 行终点 ID K99 不存在”）。
- **环检测**：检测前置关系中的循环依赖（如 K01→K02 且 K02→K01），防止后续拓扑排序死循环。
- **重复 ID**：禁止两个知识点使用同一 ID。
- **必填项缺失**：知识点名称等必填字段不可为空。

### 3.5 教师端其他功能

- 课程管理：创建 / 编辑 / 删除课程（每门课程为独立图谱空间）。
- 图谱预览：以可视化形式查看本课程当前图谱全貌。
- 学情概览（基础版）：查看本课程学生整体掌握度分布。

## 4. 数据与存储层

### 4.1 知识图谱存储

- 采用关系型数据库（MySQL）存储：**节点表 + 边表**两张核心表即构成知识图谱。
- “图谱”指”节点 + 关系”的数据组织方式，关系型数据库完全可以表达，不强制依赖图数据库。
- **Neo4j 作为可选加分项**：架构上支持平滑迁移，在文档/答辩中说明可迁移性，体现技术前瞻性。

### 4.2 多课程隔离（多租户设计）

- 所有图谱数据、学情数据均带 `course_id` 维度，实现课程间隔离。
- 学情数据按 **“学生 × 课程”** 维度存储掌握度。
- 此设计在本版即落地于数据库结构，保证后续扩展多课程时无需返工。

### 4.3 主要数据实体（概念模型）

|实体         |关键属性                                     |说明        |
|-----------|-----------------------------------------|----------|
|课程 Course  |course_id, 名称                            |一门课对应一张图谱 |
|知识点 Node   |id, course_id, 名称, 章节, 难度, 描述            |图谱节点      |
|关系 Edge    |from_id, to_id, course_id, 关系类型          |图谱边       |
|题目 Question|id, 题干, 题型, 难度, 答案                       |见第 5 章    |
|题目-知识点关联   |question_id, node_id, 主次权重               |认知诊断基础    |
|学生 Student |id, 角色                                   |用户        |
|学情 Mastery |student_id, course_id, node_id, 掌握度, 更新时间|染色与诊断的数据来源|
|学习路径记录     |student_id, course_id, 路径序列, 生成时间        |动态路径留痕    |


> 本表为概念层抽象，落地的物理表结构（字段类型、主外键、索引）详见 4.5 数据库详细设计。

### 4.4 权限模型

- 角色区分：教师（管图谱、出题、看学情）/ 学生（看图谱、做题、看路径）。
- 教师操作图谱构建与题库；学生只读图谱、读写自己的学情。

### 4.5 数据库详细设计

数据库采用 MySQL 8.0，统一字符集 `utf8mb4`，存储引擎 `InnoDB`。图谱、题库与学情相关业务表均带 `course_id` 以支持多课程隔离；学习资源表为全局共享设计（一份资源可跨课程复用），不带 `course_id`，通过关联表挂接到具体知识点。所有表统一带创建时间 / 更新时间审计字段。下表按五个功能组组织，共 14 张表。

|-------------|--------|-----------------------|---------------------------|

> **环检测**在应用层执行（导入时对 relation_type=前置 的边做有向图环路检查），不在数据库层约束。

#### 4.5.7 实体关系说明

核心关系链路如下：

- 一个 `course` 拥有多个 `kg_node` 和 `kg_edge`，构成一张知识图谱。
- `kg_edge` 通过 `from_node_id` / `to_node_id` 连接两个 `kg_node`，形成图结构。
- 一道 `question` 通过 `question_node` 关联多个 `kg_node`（含主次权重）——认知诊断的桥梁。
- 一个学生对每个 `kg_node` 有一条 `student_mastery`——染色地图的数据源。
- 学生作答产生 `answer_record`，驱动 `student_mastery` 更新，进而触发 `learning_path` 重算。
- `student_mastery` 每次更新追加一条 `mastery_snapshot`——成长曲线（7.4）的数据来源。
- `learning_path` → 多条 `learning_path_item`（拓扑排序的有序步骤）。
- `kg_node` 通过 `node_resource` 挂接多个 `resource`。

> **关于外键约束**：上表标注的 FK 为逻辑外键，体现实体关系。实际工程中是否建立物理外键约束由实现层决定（高并发场景常以应用层保证一致性），不影响数据模型设计。

## 5. 题库与知识点标注

### 5.1 题库来源策略

1. **优先**使用学校已有题库或现有题目。
1. **缺口**部分由 AI 生成题目补充。

### 5.2 知识点标注方案（核心难点）

一道题往往考查多个知识点，标注需解决”考哪些点、主次如何”的问题。采用 **“AI 辅助标注 + 人工抽检”** 的半自动方式：

**（1）AI 出题即标注（主力）**

让大模型在教师提供的知识图谱节点范围内出题，出题的同时返回所考知识点 ID 及主次权重，出题与标注一次完成、数据天然对齐。返回结构示意：

```json
{
  "question": "请说明瀑布模型的优缺点",
  "type": "简答",
  "difficulty": 2,
  "knowledge_points": [
    {"id": "K12", "name": "瀑布模型", "weight": "主"},
    {"id": "K10", "name": "软件过程模型", "weight": "次"}
  ]
}
```

**（2）存量题 AI 辅助标注**

对学校已有的未标注旧题，由 AI 读题后在本课程知识点列表中选择关联点并标注主次权重。

**（3）人工抽检校验**

AI 标注全部题目，人工只对结果抽检并修正明显错误，兼顾效率与质量。

### 5.3 题目-知识点关联字段

|字段           |作用                                           |
|-------------|---------------------------------------------|
|关联知识点 ID（可多个）|认知诊断的基础                                      |
|主 / 次权重      |做错时主知识点的薄弱嫌疑权重更高，诊断更精准                       |
|题目难度         |区分”简单题错=基础薄弱”与”难题错=正常”                       |
|题型           |客观题自动判分，参与自动诊断；主观题不参与自动诊断（见 5.4），AI 判分仅用于练习反馈|

### 5.4 软件工程课程的特殊处理

软件工程主观题（画图、论述、对比）较多，对错不易自动判定，知识点边界也较模糊。据此：

- **入口诊断题以客观题（选择 / 判断）为主**，保证可自动判分、知识点指向清晰、定位可靠。
- 主观题作为学习资源或进阶练习，**不纳入自动诊断环节**；其 AI 辅助判分仅作为练习反馈提供给学生，判分结果不写入掌握度计算（v1.1 统一口径）。
- 主 / 次权重设计在概念交叉场景下尤为关键，用于提升诊断精度。

## 6. 智能核心层

智能核心层是项目的灵魂，决定档次。按”学生使用流程”组织为四个能力。

### 6.1 学生画像（定位：你在哪）

- 学生进入后完成一套覆盖课程主要知识点的**入口诊断小测**（客观题为主）。
- 系统根据答题结果，为每个知识点计算**掌握度**并分级：已掌握 / 薄弱 / 未学。
- 结果映射到知识图谱上形成”染色地图”，学生一眼看清整体学情与薄弱区域。

**入口诊断组卷策略（v1.1 新增）**

软件工程图谱预计 50–80 个知识点，逐点出题学生不可能做完。组卷采用”抽样 + 推断”控制题量在 **20–30 题（约 15–20 分钟）**：

1. **分层抽样**：按章节分配题量，每章覆盖该章的枢纽知识点（出度 / 入度高的节点）与难度代表点。
1. **图结构推断**：利用前置关系传播掌握度——较深层节点答对，其前置链上的节点按衰减系数推断为大概率掌握，无需逐点验证；答错则触发 6.2 的回溯诊断。一份小测的信息量被图结构放大。
1. **未覆盖点标记为”未学”**，待后续答题数据进入后再更新——染色地图本身就是渐进精确的。

> 图结构推断是”图谱”真正发挥作用的第一个场景：同样 25 道题，有图谱可以覆盖判断全图，无图谱只能判断 25 个点。这一点可作为答辩讲解点。

**掌握度更新机制（v1.1 新增）**

朴素的”历史总正确率”有致命缺陷：学生初测做错、后来学会了，历史错题会永远稀释正确率，掌握度涨不上去，“动态自适应”变成假动态。因此采用**近期加权更新（EWMA，指数加权移动平均）**：

```
新掌握度 = α × 本次表现得分 + (1 − α) × 旧掌握度
```

- `本次表现得分`由本题对错、题目难度、主次权重共同决定（主考点权重高于次考点，难题答对加分多、答错扣分少）。
- `α` 为更新步长（默认 0.3，实现层可调）：保证**新表现能够覆盖旧表现**——连续答对几题，掌握度即可从”薄弱”爬回”已掌握”，学生能直观看到地图变绿。
- 掌握度分级阈值（如 ≥75 已掌握、40–75 薄弱、<40 或无数据未学）为实现层参数，待调试确定。

### 6.2 认知诊断（诊断：你为什么卡）

- 不止于”对错打分”，而是**反向定位薄弱根因**。
- 当学生在某知识点（如”二叉图/UML 图绘制”）答错，系统顺着图谱前置关系回溯，检查其前置知识点的掌握情况。
- 若前置知识点同样薄弱，判定真正根因在前置点（例：“你 UML 图掌握不好，根本原因是需求建模基础未吃透，建议先回补”）。
- 主 / 次权重参与判定：做错”主考 A、次考 B”的题，A 的薄弱嫌疑权重更高。

**回溯规则（v1.1 新增，消除实现歧义）**

|规则点     |约定                                                      |
|--------|--------------------------------------------------------|
|参与回溯的边  |仅沿**“前置”**边逆向回溯；“包含 / 相关 / 应用”边不参与根因判定（“相关”边仅用于界面上的关联提示）|
|回溯深度    |默认最多 **2 层**（实现层可配置）。超过 2 层的归因对学生已无行动指导意义，且嫌疑被稀释        |
|前置点无作答数据|标记为**“待验证”**而非直接判薄弱，向学生推送该前置点的 1–2 道验证题，用数据说话           |
|前置点均已掌握 |判定**根因即本知识点本身**，诊断结论为“前置基础没问题，是本知识点未吃透”，推荐本点资源          |
|多个前置点薄弱 |按“主次权重 × 掌握度缺口”加权排序，给出嫌疑列表（最多 3 个），首位为主要根因              |
|诊断结论呈现  |结论附带依据（“因为你在 X、Y 两题上出错，且前置点 Z 掌握度仅 35”），可解释、可追问（接 6.4）  |


> 这是项目最能体现技术深度、学生体感最强的能力——“系统真懂我卡在哪”。

### 6.3 动态自适应路径（导航：接下来学什么）

- 输入：学生薄弱点（染色地图）+ 知识点依赖关系（图谱的边）。
- 算法：在图上进行**拓扑排序**，规则为”学一个知识点前其前置必须先学”，生成有序学习路径。
- 示例：学生目标”UML 建模”，系统发现”需求分析”为前置且薄弱，生成路径：`需求分析 → 需求建模基础 → UML 概述 → 用例图 → 类图`。
- **动态自适应**：学生每次学习 / 答题后，系统实时重算剩余路径——已掌握的不再推荐，将精力导向仍薄弱处。学生每次回来看到的都是基于当前状态的最新建议。
- 路径每一步关联对应学习资源（见 7.3）。

### 6.4 AI 学习伴侣（陪伴：随时可问）

- 接入大模型，结合学生当前画像与其在图谱中的位置进行**对话式答疑与建议**。
- 区别于通用问答：能回答”为什么我要先学需求分析才能学 UML 建模”这类与个人学情绑定的问题。
- 可对路径中的某一步给出”为什么学这个、怎么学”的个性化说明。
- **防护设计（v1.1 新增）**：系统 prompt 中注入本课程知识点清单与学生当前画像，回答范围**限定在本课程图谱内**；超纲或无关问题礼貌引导回课程主题。知识点名称以图谱节点为白名单，降低幻觉（不会聊出图谱里不存在的”知识点”）。

### 6.5 AI 模型层设计（成本与数据安全）

- **不使用境外模型（如 Claude）**，避免成本与数据出网合规问题。
- 采用**可插拔模型接口**（基于 Spring AI 抽象），底层可灵活切换：
  - 开发 / 调试：国产在线 API（如 DeepSeek，成本低、中文好）。
  - 演示 / 强调数据安全：本地开源模型（如 Qwen2.5-7B，经 Ollama 部署，零 API 成本、数据完全不出本地）。
- 文档话术：“系统支持灵活切换在线 / 本地大模型，兼顾性能与数据安全，满足校内数据本地化要求。”
- AI 调用频次说明：建图谱、出题为低频操作；学生端 AI 建议相对高频，通过缓存（相同画像不重复调用）控制成本。

## 7. 体验呈现层（学生端）

### 7.1 我的知识地图

- 以 ECharts 力导向图渲染本课程完整知识图谱。
- 按掌握度染色：绿色（已掌握）/ 黄色（薄弱）/ 灰色（未学）。
- 支持点击节点查看知识点详情、关联资源、相关题目。
- 现场演示中最具视觉冲击力的一屏。

### 7.2 我的学习路径

- 一条有序的学习清单，逐步呈现”先学什么、后学什么”。
- 每步包含：知识点、对应资源链接、“为什么学这个”的说明。
- 随学情动态更新。

### 7.3 资源精准匹配

- 每个知识点挂接学习资源（视频 / 练习 / 文章），可使用公开资源（如慕课、B 站等）。
- 推荐路径时一并推送对应资源。

### 7.4 学习成长档案

- 掌握度随时间变化的曲线，呈现”成长感”。数据直接读 `mastery_snapshot` 快照表（见 4.5.5），按时间轴渲染。
- 学生层面形成个人成长记录。

### 7.5 诊断报告 / AI 对话

- 展示薄弱根因诊断结论，学生可就诊断结果与路径向 AI 追问。

## 8. 技术方案

### 8.1 技术栈选型

|层   |选型                     |理由                           |
|----|-----------------------|-----------------------------|
|前端  |Vue 3 + Vite + ECharts |主力技术栈；ECharts graph 组件满足图谱可视化|
|后端  |Spring Boot 3 + Java 21|主力技术栈，不切换语言，开发效率高            |
|数据库 |MySQL（Neo4j 可选）        |节点表+边表即可表达图谱；图数据库作加分项        |
|AI 层|Spring AI + 可插拔大模型     |在线（DeepSeek）/ 本地（Qwen）灵活切换   |
|部署  |服务器本地化部署               |满足校园网内访问与数据不出网要求             |

### 8.2 关键技术决策说明

- **不用 Python FastAPI + Neo4j 方案**：换语言在比赛周期内不划算；知识图谱在 demo 规模下用 MySQL 完全胜任，开发更快。
- **不用 Claude / 境外模型**：成本与数据合规考虑，改用国产 / 本地模型，契合赛题对数据安全与自主可控的偏好。
- **统一数据契约**：AI 生成、文件导入、手动编辑三种图谱构建方式收敛为同一内部格式，架构清晰。

### 8.3 算法要点（实现层）

- **拓扑排序**：基于图谱前置关系生成学习路径，开发前需确保图谱无环（导入时环检测兜底）。同层多个可学节点的排序依据：薄弱程度优先，其次章节顺序。
- **掌握度计算**：EWMA 近期加权更新（机制见 6.1），本次表现得分结合题目难度与主次权重；α 与分级阈值为实现层参数。
- **认知诊断回溯**：沿前置边回溯最多 2 层，主次权重参与嫌疑加权，规则细节见 6.2。
- **图结构推断**：入口诊断中沿前置链按衰减系数传播掌握度，放大小测信息量（见 6.1）。

## 9. 数据需求与来源

|数据         |用途       |来源策略                   |敏感度|
|-----------|---------|-----------------------|---|
|课程大纲 / 教学计划|构建知识图谱   |优先向学校申请；可参考公开大纲        |低  |
|单门课脱敏成绩    |学情画像、诊断验证|优先向学校申请脱敏数据            |中  |
|题库 + 知识点标注 |认知诊断     |学校题库优先，缺口 AI 生成；AI 辅助标注|中  |
|学生答题日志     |动态路径、预警  |使用合成 / 模拟数据            |高  |
|学习资源       |资源匹配     |公开资源（慕课、视频等）           |低  |

**核心策略**：项目可行性不依赖真实数据。以合成数据跑通完整功能闭环，真实数据接入作为”真实场景验证”环节加分。

## 10. 差异化亮点（对应”高级感”）

1. **AI 辅助建图谱（含量化验证）** —— 解决赛题公认最难环节（人工梳理知识点关系），并以节点召回率 / 边准确率两项指标量化呈现，把难点转化为有数据支撑的卖点。
1. **认知诊断（可解释）** —— 不是”对错打分”，而是顺着图谱反推薄弱根因，且结论附带依据、可向 AI 追问。
1. **动态自适应路径** —— 掌握度 EWMA 近期加权更新，路径随学情实时重算，而非一次性生成；新表现能覆盖旧表现，学生看得见地图变绿。
1. **AI 学习伴侣** —— 与个人学情绑定的对话式向导，区别于通用问答。
1. **统一数据契约 + 多课程架构** —— 从”单课程系统”升级为”可复制教学平台”的工程设计。
1. **本地化 / 可插拔模型** —— 成本可控、数据不出网，契合赛题数据安全导向。
